# `hypothesis/` — formal-insight ledger

This folder is the canonical home of every **formally-stated
hypothesis** about the BRIAN architecture. Each hypothesis is a single
Markdown file with YAML front-matter (machine-readable) and a KaTeX /
LaTeX body (human-readable). A sibling Lean proof file lives under
`proofs/`.

The pipeline is:

```
formal_framework.md  ──┐
                       ▼
              HypothesisRecord  ──►  emit_hypothesis_proof  ──►  proofs/H00N_*.lean
                       │                                              │
                       │                                              ▼
                       │                                      verify_lean_proof (optional)
                       │                                              │
                       ▼                                              ▼
              hypothesis/index.json   ◄───────────────  proof_status: verified
```

A discovery is the engine-authored cousin of a hypothesis; see
[`../discoveries/README.md`](../discoveries/README.md).

## File layout

```
hypothesis/
  README.md                          ← this file
  schema.json                        ← JSON Schema for HypothesisRecord
  index.json                         ← autogenerated list (don't hand-edit)
  H001_phi_monotone.md               ← one .md per hypothesis
  H002_ood_gap_decrease.md
  H003_symbolic_sparsity.md
  H004_triple_guard_soundness.md
  H005_improvement_gate_welch.md
  proofs/
    H001_phi_monotone.lean           ← one .lean per hypothesis (stub or verified)
    ...
```

## Schema

Every hypothesis file is YAML front-matter + Markdown body. The
authoritative dataclass is
[`neuroslm.discoveries.records.HypothesisRecord`](../neuroslm/discoveries/records.py):

| Field          | Type            | Meaning |
|----------------|-----------------|---------|
| `id`           | `H\d{3,}`       | Stable identifier (e.g. `H001`) |
| `title`        | `str`           | Short human title |
| `theorem_name` | `str`           | Lean-canonical name (e.g. `Brian.PhiMonotone`) |
| `status`       | enum            | `draft` ╱ `stated` ╱ `proven` ╱ `refuted` |
| `references`   | `list[str]`     | Pointers into `docs/` (e.g. `formal_framework.md §6.1`) |
| `code_refs`    | `list[str]`     | Source files realising the mechanism |
| `test_refs`    | `list[str]`     | pytest files exercising the claim empirically |
| `proof_path`   | `str` or `null` | Relative path to the `.lean` file |
| `proof_status` | enum            | `missing` ╱ `stub` ╱ `compiles` ╱ `verified` |
| `tags`         | `list[str]`     | Free-form filters |
| `created_at`   | ISO-8601        | UTC timestamp |
| `updated_at`   | ISO-8601        | UTC timestamp |

The Markdown body after the closing `---` is the **formal statement**
of the claim, with KaTeX equations (the same dialect used in
`docs/formal_framework.md`).

## Authoring a new hypothesis

Programmatic:

```python
from neuroslm.discoveries import HypothesisRecord, HypothesisStore, emit_hypothesis_proof
from pathlib import Path

store = HypothesisStore(Path("hypothesis"))
h = HypothesisRecord(
    id=store.next_id(),                              # e.g. "H006"
    title="Triple-Guard soundness",
    statement_md=r"`TripleGuard.admit` returns `admitted = True` iff all three "
                 r"sub-guards pass. Formally: $\\forall g_1, g_2, g_3.\\;"
                 r"\\mathrm{admit}(g_1, g_2, g_3) \\iff g_1 \\land g_2 \\land g_3$",
    theorem_name="Brian.TripleGuardSound",
    status="stated",
    references=["formal_framework.md §6.4", "formal_framework.md §10.2"],
    code_refs=["neuroslm/verification/triple_guard.py"],
    test_refs=["tests/training/test_rcc_bowtie_triple_guard.py"],
)
store.save(h)
emit_hypothesis_proof(h, Path("hypothesis"))         # writes proofs/H006_*.lean
store.save(h)                                        # persists proof_status="stub"
```

Manual: copy an existing `H00N_*.md`, bump the id + slug, edit the
front-matter + body, then run:

```bash
brian hypothesis emit-proofs        # regenerates any missing .lean stubs
```

## Verifying proofs

When the Lean binary is on `$PATH`:

```bash
brian hypothesis verify H001
```

Without Lean, the verifier returns a `"skipped"` verdict and
`proof_status` stays at `"stub"` — the rest of the pipeline keeps
working, but the hypothesis is not promoted to `"proven"`.

## Why YAML + Markdown + Lean

* **YAML front-matter** — easy git diffs, easy `jq`, trivially
  re-parseable from any language.
* **Markdown body** — same dialect as `docs/`, renders with KaTeX in
  every editor we care about, no proprietary format.
* **Lean** — the only choice that lets us actually mechanise the
  monotonicity / soundness claims. See
  [`../docs/formal_framework.md`](../docs/formal_framework.md) §10
  for the planned Brian Lean library namespace.
